# Journal -> RemCard Migration Plan

## Цель
- Оставить журнал максимально простым:
  - только регистрация пациента на койку (интейк).
- Перенести клиническую работу в RemCard:
  - ИВЛ, трансфузии, перевод/исход, дальнейшая динамика.
- Сохранить единую БД и безопасную миграцию без потери данных.

## Принципы
- Одна база данных (`rao_journal.db`) остается источником истины.
- Изменения делать поэтапно, без "big bang" переписывания.
- Сначала совместимость данных, потом UI-упрощение.
- Любая миграция БД только additive-first (добавляем, не ломаем).

## Целевое разделение ответственности
- Журнал:
  - создание/редактирование базовой карточки поступления;
  - ФИО, ИБ, пол/возраст, время поступления, источник/профиль;
  - диагноз МКБ-10 + ручной текст;
  - управление занятостью койки.
- RemCard:
  - ИВЛ, трансфузии, процедуры, события, исход;
  - таймлайн случая до завершения;
  - перевод/смерть + автоматическое освобождение койки по таймеру.

## Этап 0. Архитектурная стабилизация DB-слоя
- Вынести общий runtime-движок БД в единый core:
  - lock/reconnect/backups/auto-recovery/outbox/local-replica.
- Оставить 2 тонких адаптера API:
  - `RemCardDBAdapter`, `JournalDBAdapter`.
- Цель этапа:
  - не менять внешний контракт существующего кода сразу,
  - убрать дублирование логики двух DB-менеджеров.

### Готово, если
- DB-восстановление/бэкапы/реплика/очередь работают одинаково для обоих контуров.
- В коде нет дублирующейся "инфраструктурной" логики между менеджерами.

## Этап 1. Упрощение формы журнала (UI)
- Удалить из `PatientForm` вкладки:
  - "Трансфузии", "ИВЛ", "Перевод/Исход".
- Оставить:
  - "Общие данные", "Диагнозы".
- Сохранение в журнале:
  - только поля интейка и диагноза + операции (если оставляем операции в журнале на переходный период).

### Готово, если
- Открытие/создание карточки в журнале не показывает клинические вкладки.
- Создание пациента/поступления работает как раньше по базовым полям.

## Этап 2. Перенос клинических сущностей в RemCard UI
- ИВЛ:
  - редактирование полностью из вкладки ИВЛ RemCard.
- Трансфузии:
  - учет через назначения/процедуры и записи событий в RemCard.
- Перевод/исход:
  - закрытие случая через RemCard timeline/events.

### Готово, если
- Пользователь не возвращается в журнал для клинических действий.
- Все изменения отражаются в общей БД и видны обеим ролям.

## Этап 3. Авто-освобождение койки после исхода
- Ввести автоматический таймер освобождения койки:
  - например 30 минут после события "Переведен"/"Умер".
- До истечения таймера:
  - редактирование ограничить (или пометить как "closing window").
- После истечения:
  - admission переводится в архивный статус,
  - койка освобождается автоматически.

### Готово, если
- Исход из RemCard гарантированно завершает кейс и освобождает койку через таймер.
- Ручное вмешательство не требуется.

## Этап 4. Архив и статистика
- В архиве (RemCard) оставить выборку по периоду как главный фильтр.
- Кнопки:
  - "Отчет и статистика" -> заменить на "Статистика" (пока disabled placeholder).
  - "Сформировать графики" вынести в отдельную кнопку рядом.
- Логика:
  - статистика и графики используют текущую архивную выборку (тот же диапазон дат/фильтры).

### Готово, если
- Период архива и период статистики/графиков всегда синхронизированы.
- Пользователь не вводит период дважды в разных местах.

## Этап 5. Button-lock между ролями (добавление пациента)
- Добавить "кнопка-лок" на действие "занять койку/добавить пациента":
  - если врач начал создание на койке N, у медсестры эта кнопка неактивна;
  - и наоборот.
- Реализация:
  - файловый lock (по аналогии role lock), но ключ = `bed_number` + action type.
  - heartbeat + stale-timeout.
- При закрытии формы/сохранении/отмене lock снимается.

### Готово, если
- Нельзя одновременно открыть форму создания одного и того же пациента с двух ролей.
- При аварийном закрытии lock корректно протухает.

## Этап 6. Модель данных для "легкого добавления полей"
- Ввести формальный слой миграций:
  - таблица `schema_migrations(version, applied_at, note)`.
- Политика:
  - сначала `_ensure_column`/новая таблица/индекс;
  - затем read/write в коде с дефолтами;
  - только потом включение в UI.
- Для расширяемости интейка:
  - добавить `admissions.intake_extra_json` (или отдельную intake-таблицу),
  - хранить новые редко используемые поля без массовых рефакторов.
- Пример:
  - добавление "вес при поступлении" делается миграцией + один маппер поля в UI.

### Готово, если
- Новое поле можно добавить без каскадной переделки сервисов/DAO/UI.
- Старые записи читаются без ошибок.

## Этап 7. Депрекация старых журнал-сценариев
- Удалить из журнала неиспользуемые repositories/services для ИВЛ/трансфузий/перевода.
- Оставить только:
  - intake,
  - bed occupancy,
  - быстрый доступ к архиву (если нужен).

### Готово, если
- В журнале нет "мертвого" функционала.
- Поддержка и тестирование упрощены.

## Технические риски и контроль
- Риск: регрессия при объединении DB-ядра.
  - Контроль: интеграционные smoke-тесты startup/read/write/recovery.
- Риск: гонки при button-lock.
  - Контроль: lock-файлы + heartbeat + stale cleanup.
- Риск: рассинхрон периода архива и графиков.
  - Контроль: единый объект фильтра периода, передача как single source.

## Предлагаемый порядок работ (реальный)
1. Этап 0 (единый DB-core + адаптеры).
2. Этап 5 (button-lock), чтобы сразу убрать конфликтные операции.
3. Этап 1 (упрощение журнала UI).
4. Этап 2-3 (перенос клиники и авто-завершение кейса).
5. Этап 4 (архив/статистика UX).
6. Этап 6-7 (миграции и финальная чистка).
